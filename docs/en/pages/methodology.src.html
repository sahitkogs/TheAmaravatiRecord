HOW THE NUMBERS ARE MADE

Methodology

The data pipeline, sources, classification approach, and error bars behind every report in The Amaravati Record — documented so any reader can check the arithmetic.

LAST UPDATED: APRIL 10, 2026  |  VERSION 1.0

The Amaravati Record is built on a single working principle: every number we publish must be traceable to a specific line in a public dataset, and every transformation between that line and the published headline must be reproducible from code in a public repository. This page documents the pipeline. If you disagree with a finding, this is where you look to find the disagreement.

1. Source data

The primary source for our land-pooling investigations is the APCRDA Land Pooling Scheme portal, an official Government of Andhra Pradesh site that publishes the reconstituted plot-level beneficiary roll for the twenty-six capital-region villages.

Supplementary sources used during classification:

  • Community surname lists. Nineteen independent community websites, blogs, caste-association registers, and Telugu reference sites. Each source is retained as-scraped, URL-backed, and traceable. The merged corpus contains over 5,500 surnames.
  • MyNeta candidate database. Election commission-backed candidate declarations, used as a cross-reference for caste attribution of public figures.
  • APCRDA village boundary GeoJSON. Public government shapefile used for the map views.

2. Scraping

The APCRDA portal is scraped by a documented Python scraper (scrape_apcrda_lps.py) that respects robots.txt, runs at rate limits compatible with the portal's terms, and does not require authentication. The scraper outputs a raw CSV of plot records, retained verbatim before any cleaning. This raw extract is the ground truth against which all downstream numbers are audited.

Raw extract: ~95,645 records. These include duplicate entries introduced by the portal's paginated rendering, occasional malformed rows, and records for plots that were re-issued after boundary adjustments.

3. Cleaning and deduplication

The raw extract passes through a cleaning pipeline (build_report.py) that:

  • Normalizes village names (e.g. NOWLURU, Nowluru, Navuluru → Nowlur) using an explicit mapping table, which is version-controlled so every reader can see it.
  • Deduplicates plot records on a composite key of village + plot identifier.
  • Excludes records that the portal marks as company-owned (containing keywords like LIMITED, PRIVATE, TECHNOLOGIES, etc). Individual-owned plots are our unit of analysis; corporate allocations are reported separately in a dedicated section.

After this pass: 47,993 unique individually-owned plots across 26 villages. This is the number used in every published statistic unless otherwise stated.

4. Ground-truth surname corpus

Parallel to the APCRDA pipeline, we build a hand-curated surname-to-caste directory from nineteen independent sources, each with a URL, an extraction script, and a retention copy of the source page. The corpus goes through the following stages:

  • Extraction: names are extracted from community websites, caste-register pages, and election-candidate records.
  • First-name detection: a frequency analysis detects 351 strings that are first names rather than surnames, which are flagged and excluded from the surname corpus so they do not corrupt lookups.
  • Conflict reconciliation: surnames that appear across multiple sources with different caste attributions are reconciled using a documented priority order (primary source beats aggregator, URL-backed beats undated, regional source beats pan-Indian).
  • Retention: the final corpus is published as explorer/surname_ground_truth.csv under ODbL, with the full source-to-surname trail preserved.

Final corpus size: 5,548 surnames across 50+ caste categories, each with URL-backed evidence.

5. Per-name classification

Every beneficiary name from the cleaned APCRDA roll is classified through a two-layer pipeline:

  1. Deterministic lookup. The name is parsed into surname and given-name components and the surname is looked up in the ground-truth corpus. If a confident match is found, the classification is stamped with the source URL for the match.
  2. Model fallback. For names where the surname is missing, ambiguous, or where the name string contains only a given name, the full name is passed to Google Gemini 2.5 Flash with a structured prompt asking for the classification, a confidence level, and a brief reason. This handles approximately the residual few thousand names per run.

The combined pipeline achieves 99.6% classification rate on the 47,993-plot dataset. The residual 0.4% are explicitly reported as Unknown rather than silently assigned.

6. Cross-checking and uncertainty

For every name where both layers produce an output (i.e. names whose surname is in the ground-truth corpus and whose full name is classifiable by the model), we record whether the two agree. The agreement rate is the primary reliability measure of the pipeline and is published on the dashboard's methodology tab for every run.

Individual-level attributions carry a measured uncertainty. Aggregate community statistics (the percentages published on the home page and in the reports) are reported with explicit error bars derived from the disagreement rate applied as a monte-carlo resampling over the classification buckets. The margin of error on the Kamma 57.4% headline figure is documented on the interactive dashboard.

7. Privacy and masking

Unmasked names are held privately on the publisher's local machine and are never published. Every public report masks:

  • Farmer given names (replaced with ***)
  • Plot identifiers (village code preserved; specific plot numbers redacted)

Aggregate statistics by community, village, and plot category are unaffected by the masking. The published reports and datasets under our ODbL data license reflect the masked state; the unmasked state exists only privately and is not shared.

8. The pipeline in code

Every stage above is implemented in documented Python scripts in the project repository:

  • scrape_apcrda_lps.py — step 1, portal scraper
  • build_report.py — steps 3+8, cleaning, deduplication, aggregation, and report generation
  • caste_classifier_gemini.py — step 5, per-name Gemini classification
  • explorer/build_surname_explorer.py — step 4, surname explorer build
  • utils/name_utils.py — shared name-parsing helpers
  • tests/test_build_report.py, tests/test_data_validation.py — unit tests and validation

9. Reproducibility

Any reader with a Gemini API key and a Python environment can:

  1. Clone the repository
  2. Run python scrape_apcrda_lps.py to fetch fresh APCRDA data (or use the committed extract for version-locked reproducibility)
  3. Run python caste_classifier_gemini.py to regenerate per-name classifications
  4. Run python build_report.py to produce the full interactive dashboard and broadsheet outputs
  5. Compare against the published version — the only expected differences are those introduced by (a) model stochasticity on the residual few thousand Gemini-classified names and (b) subsequent APCRDA portal updates

10. What this methodology does not claim

  • It does not claim to be a census. Caste classification from name alone is an approximation.
  • It does not claim that individual beneficiaries necessarily identify with the community attributed to them. People marry, migrate, adopt, and reclassify; a name is not an identity document.
  • It does not claim that any particular individual's inclusion in the beneficiary roll implies wrongdoing. The Land Pooling Scheme is a lawful programme with public eligibility criteria; this publication reports the pattern of allocation, not verdicts on individual cases.
  • It does not claim neutrality on methodological choices. Choices were made — about what counts as a "surname", what counts as an "individual" plot, what error bound counts as "publishable". All of those choices are stated explicitly so readers can disagree with them openly and propose alternatives in corrections.

11. AI usage in this pipeline

The pipeline uses Google Gemini 2.5 Flash for the per-name classification fallback, as described above. It does not use language models to write prose, headlines, analysis, or editorial commentary. See the full AI disclosure for details on how model outputs are cross-checked against the hand-curated corpus before becoming part of any published statistic.
